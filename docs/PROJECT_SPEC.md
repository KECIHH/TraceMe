# Project Spec

TraceMe（迹遇）是一个私有部署的旅行规划网站（Private Travel Planner），目标是在自有服务器上通过域名和 HTTPS 安全访问，用于个人、家庭或可信小范围成员管理旅行资料。

它不是公开 SaaS，不提供公众注册、公开社区、公开分享、在线支付或商业化能力。当前默认只有 seed 创建的管理员账号；如果后续增加后台创建用户，也应保持小范围使用。在完成多人权限系统、账号治理、滥用防护和合规审查前，不建议开放公众使用。

## 项目目标

- 支持通过 `https://travel.example.com` 这类域名访问。
- 默认强制登录，未登录用户不能访问核心页面。
- 使用 Docker Compose 进行可维护的生产部署。
- 使用 SQLite、上传文件目录和备份目录的持久化 volume。
- 避免把 `.env`、数据库、上传文件、备份文件暴露为静态资源。
- 提供可验证的 lint、typecheck、unit test、e2e test、build 和 Docker 部署路径。

## 功能模块

- 账号与会话：登录、退出、修改密码、会话清理、受保护页面鉴权。
- Dashboard：旅行数量、近期旅行、预算、文件、笔记和备份入口。
- 旅行规划：创建、查看、编辑、归档、删除、搜索、状态和日期管理。
- 目的地：国家、地区、时区、抵离日期和备注。
- 地点库：景点、餐厅、酒店、车站、机场、商店、医院、使领馆、交通点等。
- 行程日历：按日期生成天数、管理每日信息、行程项、排序、状态和冲突提示。
- 今日模式：展示当天或最近行程、住宿和文件入口。
- 路线规划：路线计划、交通方案、权重预设、评分、选择和删除。
- 准备清单：自定义清单项、状态切换、完成率和模板清单生成。
- 美食与住宿：基于地点扩展餐厅详情和住宿预订详情。
- 预算：分类预算、支出记录、汇率换算和剩余额度。
- 文件：上传、编辑、下载、删除、大小限制、类型校验和存储目录隔离。
- 笔记：攻略、备忘、来源链接、标签，以及 AI 结果保存为笔记。
- AI 助手：mock 或 OpenAI provider，生成草稿、敏感信息提示、历史摘要保存。
- 导出：单旅行 JSON、Markdown、HTML 导出。
- 备份：系统级 zip 备份，包含数据库快照和上传文件，不包含 `.env`。
- 设置中心：个人资料、密码、AI 开关、备份管理、系统信息。
- 部署：Dockerfile、docker-compose.yml、环境变量校验、健康检查和反向代理文档。

## 数据模型概览

- `User` / `Session`: 管理员账号、密码 hash、登录会话。
- `Trip`: 旅行主表，保存标题、状态、日期、预算、目的地摘要。
- `Destination`: 旅行目的地。
- `Place`: 地点库，可扩展美食和住宿。
- `FoodDetail` / `StayDetail`: 餐厅与住宿详情。
- `ItineraryDay` / `ItineraryItem`: 行程日期和当天行程项。
- `RoutePlan` / `TransportOption`: 路线规划和交通候选方案。
- `ChecklistItem`: 准备清单。
- `CategoryBudget` / `Expense`: 分类预算和支出。
- `Document`: 上传文件元数据，磁盘文件在 `storage/uploads`。
- `Note`: 笔记和 AI 草稿保存结果。
- `AiConversation`: AI 历史摘要和脱敏提示。
- `AiPlanDraft`: AI 结构化旅行草稿。
- `AppSetting`: 应用设置。
- `BackupRecord`: 备份文件记录，磁盘文件在 `storage/backups`。

## 安全与访问策略

- 默认强制登录。
- 暂不开放注册，管理员通过 seed 创建。
- cookie 必须 `httpOnly`、`sameSite=lax`；HTTPS 域名访问时必须 `secure`，临时 HTTP IP 测试时允许不启用 `secure`。
- 登录失败有基础限流。
- API 不返回 `passwordHash`。
- `/api/health` 不返回密钥、数据库绝对路径、用户信息、文件路径或完整环境变量。
- 上传文件和备份文件不放入 `public`，必须通过鉴权 API 访问。
- AI Key 只在服务端环境变量中读取。
- 默认 `robots.txt` 阻止索引，页面 meta 默认 noindex。

## 非目标

- 不提供公开注册。
- 不做公开社区。
- 不做公开分享。
- 不做在线支付。
- 不做商业化。
- 不提供多租户权限矩阵。
- 不把上传文件、备份文件或数据库作为静态文件暴露。

## 未来公开化前置条件

如果未来改为公开网站，至少需要补充：

- 隐私政策和用户协议。
- 完整账号系统、找回密码、邮箱验证和强密码策略。
- 多人权限系统和资源隔离。
- 滥用防护、限流、审计日志、风控和内容治理。
- 合规审查、数据保留与删除机制。
- 重新配置 robots、SEO、站点地图和公开分享边界。
